# AGENTS.md

## 项目目标

这是一个用于收集和展示采集数据的全栈项目。

- 后端使用 Go + Gin。
- 前端使用 Next.js + TypeScript。
- 当前阶段优先保证两个项目能独立启动、联调成功，并保持结构简单可扩展。

## 工作区结构

- `backend/`: Go + Gin API 服务。
- `frontend/`: Next.js 前端看板。
- `README.md`: 启动方式和 API 说明。

## 后端约定

- 使用 Go + Gin 编写 HTTP API。
- 开发初期可以使用内存数据，后续再接入 PostgreSQL。
- API 路径统一放在 `/api` 下。
- 健康检查接口使用 `/health`。
- JSON 字段优先使用 camelCase，方便前端消费。
- 保持处理逻辑简单清晰，不提前引入复杂分层，等功能增长后再拆分 `handler/service/repository`。

## 前端约定

- 使用 Next.js App Router。
- 使用 TypeScript。
- 页面文案默认使用简体中文。
- 初期优先使用原生 CSS，避免过早引入 UI 框架；后续需要组件库时优先考虑 shadcn/ui。
- 与后端联调时默认后端地址为 `http://localhost:8080`。

## 开发与验证

后端：

```powershell
cd backend
go mod tidy
go run .
```

前端：

```powershell
cd frontend
npm install
npm run dev
```

验证：

- 后端健康检查：`GET http://localhost:8080/health`
- 数据列表：`GET http://localhost:8080/api/data-points`
- 前端页面：`http://localhost:3000`

## 协作偏好

- 默认使用简体中文沟通。
- 修改代码前先理解现有结构，尽量小步、聚焦地变更。
- 遇到上游命令因工作目录错误导致失败时，不需要停下解释，直接在正确目录重新执行。
- 遇到包含“上游”“可重试”“Agent”“出错”等字样的上游/Agent/网络/本地环境不明错误时，不要停止等待用户输入，直接重试当前操作。
- 遇到“模型服务商”“可重试”“模型服务商暂不可用”“USER_LLM_PROVIDER_GATEWAY_ERROR”“service temporarily unavailable”“gateway_unavailable”等模型服务商临时不可用或 503 网关错误时，不要停止等待用户输入，稍后或立即重试当前操作；必要时可重发消息或切换可用服务商。
- 遇到“USER_LLM_PROVIDER_NETWORK_ERROR”“无法连接模型服务商”“模型服务商连接失败”等网络连接类错误，或网络、DNS、Base URL、接口路径、代理连接异常导致的模型服务商调用失败时，无需等待用户确认，直接重试当前操作；必要时可重发请求或切换可用服务商。
- 当第三方依赖、包、模块或命令缺失时，可以直接按项目包管理方式下载或安装依赖（例如 Go module、npm package 等），无需停下询问，也不要切换到兜底工具绕过；安装后继续当前任务，并在汇报中说明安装了什么。
- 团队任务仍为 `in_progress`、负责人处于 idle 且尚未提交明确最终回报时，不得停止推进或视为完成；Team lead 应主动重新唤醒负责人继续执行，无需等待用户提醒。只有实现完成、必要验证完成、任务板标记为 `completed`，并提交包含修改文件、验证结果、遗留问题等内容的最终回报后，任务才算真正完成。
- 在不违反平台/system 指令与 Team Governance 等更高优先级规则的前提下，用户发布的任务只有在用户明确要求停止/暂停，或负责助手明确报告任务已完成后才可停止推进；idle、无回应、任务板未更新，以及 `UNKNOWN_UPSTREAM_ERROR`、响应解码失败、网络错误、模型服务商临时错误等均不构成停止条件，必须持续重试、主动唤醒或合理重新分配。收到完成报告后仍需验收，验收失败应继续推进直至满足要求。
- 当任一请求发出后超过 1 分钟仍未收到响应时，视为连接或请求异常；不要继续无期限等待，应主动断开当前连接并重新连接，然后自动重新发起同一请求或提问，无需等待用户确认，同时避免因重复请求产生不可逆的重复操作。
- **错误后任务自检与自动恢复（最高执行规则）**：每次遇到任意工具、命令、Agent、网络、DNS、模型服务商、Base URL、HTTP 4xx/5xx、构建、测试、响应解码或本地环境错误后，必须立即检查当前用户任务、任务板和验收状态。只要用户需求尚未实现、验证未通过、任务仍为 `pending`/`in_progress`，或没有明确完成验收，就必须自动重试失败操作、修复原因并重新执行后续步骤；不得因报错、空闲、超时或模型服务商不可用而停止、等待用户提醒或错误宣告完成。对可能产生不可逆副作用的操作，重试前应先核验当前结果以避免重复创建。
- **断点续跑无需确认（最高执行规则）**：当任务执行过程中出现任何报错、暂停、中断、超时、Agent idle、上游异常或本地环境问题时，不得停下等待用户确认；必须从上一个可继续的断点自动恢复、自动重试并继续执行，直到任务真正完成或用户明确要求停止。
- **业务数据与验收隔离（最高数据保护规则）**：`backend/data/app.db`、其 WAL/SHM 文件和 `backend/uploads/` 默认均视为用户生产数据。构建、测试、浏览器验收和脚本不得删除、模糊匹配删除、覆盖、重置或清空这些路径中的任何内容；不得使用真实业务 API 批量清理“验收数据”。需要写入/删除测试数据时，必须通过 `SQLITE_PATH`、`UPLOAD_DIR` 等配置启用独立临时数据库和临时上传目录，并在该隔离环境内清理。删除用户文件必须采用可恢复的回收站/软删除策略，任何永久物理删除均须用户明确授权。
- 不主动提交 Git commit，除非用户明确要求。

# 跨业务团队通用执行规则

版本：1.1
编码：UTF-8
适用范围：当前团队、后续加入的所有成员，以及采用本规则的其他团队。

## 一、基本原则

1. 任务必须持续推进到完成、明确阻塞或收到暂停指令，不得因单次模型、服务商、网络或解析错误终止整项任务。
2. 发生可恢复错误时，必须保留已有文件、日志、状态、任务进度和已验证结论，从最近成功节点继续；禁止删除成果后从头重做。
3. 恢复顺序为：盘点已完成内容和现有产物，定位最近成功节点，记录错误，有限退避，重试未完成步骤，验证结果，再继续后续步骤。
4. 在不违反 DR-10M「十次闸门」的前提下，以尽快恢复和完成任务为优先，不因 token 或 API 调用费用降低必要的恢复力度。
5. 不得形成无上限、无退避或多成员同时触发的重试风暴。

## 二、可恢复故障范围

下列错误一律视为可恢复故障，并按断点恢复流程处理：

1. 模型或服务商错误，包括临时限流、HTTP 429、服务繁忙、路由异常、临时鉴权异常和服务暂不可用。
2. 网关或上游不可用错误，包括 USER_LLM_PROVIDER_GATEWAY_ERROR、HTTP 502、HTTP 503、HTTP 504、network_error、upstream_unavailable、service_unavailable、gateway_unavailable，以及提示“网络连接暂时异常，请稍后自动重试”“service temporarily unavailable, please retry”等同义错误。
3. 服务商网络错误，包括 USER_LLM_PROVIDER_NETWORK_ERROR、HTTP 请求发送失败、连接拒绝、连接重置、连接中断和流式响应中断。
4. DNS 或域名解析错误，包括 getaddrinfo、ENOTFOUND、EAI_AGAIN、名称解析失败和名称解析超时。
5. 服务端错误，包括 HTTP 5xx、网关错误和临时后端不可用。
6. 超时错误，包括连接超时、请求超时、首字超时、总响应超时和读取超时。
7. 响应数据或解析错误，包括 UNKNOWN_UPSTREAM_ERROR、响应体解码失败、error decoding response body、格式异常、截断响应和临时协议不兼容。
8. 其他具有瞬时性、重试后可能恢复，且未被明确判定为永久失败的模型、HTTP、网络、路由或服务商错误。

## 三、90 秒无内容响应策略

1. 从服务商或模型请求成功发出时立即开始计时。
2. 发出请求后 90 秒内未收到任何有效响应内容时，必须立即取消、终止或丢弃本次请求，并从最近有效断点重新发送尚未完成的部分。
3. 如果在 90 秒内已经收到任何有效响应内容，则视为通过本次 90 秒闸门；不再仅因请求总时长超过 90 秒而取消，可以继续等待并正常处理后续响应。
4. 本策略正式替代旧版“首字超过 30 秒提前取消”及“请求总时长达到 90 秒仍未完成即取消”的规则。新的判定标准是 90 秒内是否收到任何有效响应内容，而不是请求是否在 90 秒内全部完成。
5. 取消、失败和无内容超时不得导致整项任务重做；重发前必须复用已经生成并验证的产物。
6. 每次重试应采用合理的有限退避，并记录错误类型、发生时间、重试次数、最近成功节点和恢复结果。
7. 若多次有限重试后仍无法恢复，应保留可续接状态，报告断点、阻塞证据和下一次恢复入口，不得伪报完成。
8. 90 秒无内容响应策略必须服从 DR-10M。接近共享调用上限时，应先等待滚动窗口释放，再重发请求。

## 四、断点续做规范

1. 重试前先盘点：任务状态、已完成事项、已有文件、有效成果、当前进度、未完成步骤、最后成功操作、最近验证断点和错误证据。若任务位于 Git 工作区，必须同时检查 git status 和 git diff，确认已有改动后仅续做未完成部分。
2. 已完成且验证通过的步骤不得重复执行，除非其结果已经失效或明确需要复核。
3. 若执行流中断，应复用原任务目录、草稿、日志、缓存和中间产物，从最近可靠断点继续。
4. 修改已有产物前应先读取并理解当前内容，只做完成任务所需的增量修改。
5. 恢复后必须验证产物完整性和任务状态，并在报告中说明从哪个断点恢复。

## 五、临时文件与产物隔离

1. 临时文件必须写入为当前任务创建的专用子目录，不得散落在工作区根目录或污染业务目录。
2. 正式产物必须写入任务指定位置；诊断日志、缓存、重试记录和中间文件不得混入正式交付目录，除非任务明确要求。
3. 不得覆盖或删除用户已有文件，除非获得明确授权并具备可验证的备份或回退路径。
4. 临时目录和文件名应能识别所属任务或时间窗口，便于断点恢复和审计。

## 六、DR-10M「十次闸门」

1. DR-10M 启用时，全团队共享一个滚动 60 秒窗口，窗口内最多允许 10 次成功的模型或服务商调用。
2. 上限是全团队合计 10 次，不是每名成员、每个 Team、每个模型或每个 Provider 各有 10 次。
3. 仅成功的模型或服务商调用计入额度。失败、超时或被取消的调用不计成功数，但仍必须避免重试风暴。
4. 请求和重试预算由团队负责人统一协调。成员发起调用前应服从负责人的预算安排；接近上限时等待滚动窗口释放。
5. 不得通过切换 Team、Provider、模型、账号，或新增、替换团队成员规避共享上限。
6. DR-10M 优先于“效率优先”和快速重试策略。两者冲突时，必须先遵守十次闸门，再在剩余预算内选择最高效的恢复方式。
7. DR-10M 启用期间，应减少无必要的模型调用，优先使用已有产物、确定性本地工具和一次可完成的批量操作。
8. 用户说“取消 DR-10M 规则”或“停用十次闸门”时，暂停执行 DR-10M。
9. 用户说“启用 DR-10M 规则”或“启用十次闸门”时，恢复执行 DR-10M。
10. 除非用户明确使用上述停用口令，否则不得自行暂停、绕过或放宽 DR-10M。

## 七、HTR-90「九十秒硬闸」

1. HTR-90 启用时，每次模型、Provider、网关或上游请求从发出到完成最多允许 90 秒。达到 90 秒仍未完成时，必须立即取消当前请求；已收到首字或部分响应不构成例外。
2. 首个有效响应内容超过 30 秒仍未到达时，应在 30 秒阈值提前取消，不必等待到 90 秒。
3. 若底层工具、运行时或服务不支持主动取消，应在恢复控制权后抛弃迟到结果，不得将超过 90 秒才完成的响应视为有效成果；随后从最近成功节点重试。
4. 典型判定：总用时 316 秒、首字 16.1 秒的请求仍不合格，因为总用时超过 90 秒；总用时 15 秒且首字不超过 30 秒的请求符合 HTR-90 时间要求。
5. 每次超时必须记录：执行成员、任务标识、总用时、首字时间、最近有效断点、取消或抛弃方式，以及后续重试结果。
6. 重试前必须检查任务状态、已生成文件、有效成果和最近验证断点；Git 工作区还必须检查 git status 和 git diff。只续做未完成部分，禁止覆盖、删除或重复生成已验证成果。
7. HTR-90 与有限退避、断点恢复并行执行：超时后先保留状态和产物，再有限退避，并从最近成功节点重试。
8. 所有重试仍服从 DR-10M「十次闸门」。接近全团队共享调用上限时，应等待滚动窗口释放；不得为了满足 HTR-90 而规避 DR-10M 或形成重试风暴。
9. HTR-90 当前启用。除非收到明确的规则变更指令，否则所有当前及未来成员均必须执行。

## 八、负责人和成员职责

1. 团队负责人负责协调全团队调用预算、重试时机、任务优先级和跨成员恢复安排。
2. 成员必须及时报告调用需求、可恢复错误、已消耗的成功调用、最近成功节点和重试结果。
3. 成员不得在接近 DR-10M 上限时自行并发重试，也不得通过创建新成员绕过预算。
4. 新加入的成员在执行任何任务前必须阅读并遵守本规则；负责人应将规则和当前 DR-10M 启停状态同步给未来成员。
5. 任务交接时必须同时交接已有产物、当前进度、错误记录、最近成功节点、未完成步骤和调用预算状态。

## 九、执行检查清单

1. 是否识别并记录了错误类型？
2. 是否将该错误按可恢复故障处理？
3. 是否保留了已有产物和当前状态？
4. 是否从最近成功节点继续，而非整项重做？
5. 是否执行首字 30 秒提前取消和总用时达到 90 秒绝对硬取消/抛弃？
6. 若底层无法主动取消，是否在恢复控制权后抛弃了迟到结果？
7. 重试前是否检查任务状态、已有文件、有效成果和最近验证断点；Git 工作区是否检查 git status 和 git diff？
8. 是否记录了超时成员、任务、总用时、首字时间、断点和重试结果？
9. 是否使用有限退避并避免重试风暴？
10. 临时文件是否只位于专用子目录？
11. DR-10M 当前是否启用？
12. 最近滚动 60 秒内全团队成功调用数是否低于 10 次？
13. 是否已由负责人协调本次请求或重试预算？
14. 是否避免通过 Team、Provider、模型、账号或成员变更规避限制？
15. 完成后是否验证产物并报告结果、残余风险和可续接断点？
